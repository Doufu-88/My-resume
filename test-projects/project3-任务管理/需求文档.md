# 项目三：任务管理 API 系统

## 需求文档

### 1. 项目概述
构建一个面向团队协作的任务管理 RESTful API 系统，支持任务创建、分配、状态流转、评论、附件上传、统计报表等功能。

### 2. 功能需求

#### 2.1 项目管理
- 支持创建项目，包含项目名称、描述、开始/结束日期、项目状态（未开始/进行中/已结束）
- 项目成员管理（添加/移除成员，设置角色：项目管理员、开发者、观察者）
- 项目管理员可修改项目信息、管理成员
- 结束日期不能早于开始日期
- 项目删除为软删除（标记 deleted=1），30天后自动物理删除

#### 2.2 任务管理
- 任务包含：标题、描述、优先级（P0/P1/P2/P3）、状态、指派人、截止日期、标签
- 任务状态流转：待处理 → 进行中 → 待审核 → 已完成
- 状态流转规则：只能按顺序流转，不可跳过，但可从任意状态回退到"待处理"
- P0 任务创建后自动通知所有项目成员（站内信+邮件）
- 指派人只能有一个，修改指派人需记录变更历史
- 任务支持父子关系（子任务完成后，父任务才能完成）
- 逾期任务（超过截止日期未完成）列表中标红

#### 2.3 评论系统
- 任务详情页支持评论功能
- 评论支持 @ 项目成员，被 @ 的成员收到通知
- 评论可编辑（仅作者可编辑，编辑后显示"已编辑"标记）
- 评论可删除（作者或项目管理员可删除）
- 评论按时间倒序排列

#### 2.4 附件上传
- 支持上传附件关联到任务
- 单个附件最大 50MB
- 支持格式：jpg/png/gif/pdf/doc/docx/xls/xlsx/txt/zip
- 附件上传后返回下载链接，有效期为24小时
- 过期附件需重新生成下载链接
- 删除附件需二次确认

#### 2.5 看板视图
- 按任务状态分为4列展示（待处理/进行中/待审核/已完成）
- 支持拖拽任务卡片改变状态
- 支持按优先级、指派人、标签筛选
- 看板视图实时更新（WebSocket 推送）

#### 2.6 统计报表
- 项目维度统计：任务总数、各状态任务数、延期任务数
- 成员维度统计：各成员的完成任务数、平均处理时长
- 时间维度统计：按周/月统计任务创建和完成趋势
- 报表支持导出为 Excel

### 3. 非功能需求
- 接口遵循 RESTful 规范
- 使用 WebSocket 推送实时更新（看板、通知）
- API 版本管理（/api/v1/...）
- 敏感操作（删除任务/项目、移除成员）需记录审计日志
- 分页接口默认 pageSize=20，最大100

## 已知 Bug 列表（用于测试实操）

### Bug #1：WebSocket 断线后状态丢失（严重）
- **模块**：看板视图
- **现象**：WebSocket 断线重连后，看板卡片位置被重置，期间其他用户的更新丢失
- **复现步骤**：打开看板 → 断开网络30秒 → 重连，期间的更新不显示
- **预期**：重连后自动获取增量更新，补充丢失的数据
- **实际**：重连后重新拉取全量数据，但显示的仍是重连时刻的快照

### Bug #2：任务状态流转可跳过审核环节（严重）
- **模块**：任务管理 - 状态流转
- **现象**：通过直接调用 API，可将任务从"进行中"直接改为"已完成"，跳过"待审核"
- **复现步骤**：调用 `PUT /api/v1/tasks/{id}` 直接将 status 从 "in_progress" 改为 "completed"
- **预期**：后端应校验状态流转合法性，拒绝非法状态转移
- **实际**：后端未校验状态流转路径，直接更新成功

### Bug #3：并发修改指派人导致数据不一致（严重）
- **模块**：任务管理 - 指派
- **现象**：A和B两个管理员同时修改同一任务的指派人，最终结果不可预期
- **复现步骤**：管理员A将任务指派给用户X的同时，管理员B将同一任务指派给用户Y
- **预期**：使用乐观锁或悲观锁，后者提示冲突
- **实际**：无并发控制，以最后写入的为准，但变更历史记录了两条

### Bug #4：附件删除确认可被绕过（中等）
- **模块**：附件上传 - 删除
- **现象**：前端显示二次确认对话框，但直接调用 DELETE API 可绕过确认直接删除
- **复现步骤**：用 Postman 直接调用 `DELETE /api/v1/attachments/{id}`
- **预期**：后端也应有二次确认机制或延迟删除
- **实际**：二次确认仅在前端，后端直接执行物理删除

### Bug #5：@ 通知在评论编辑后重复发送（中等）
- **模块**：评论系统
- **现象**：编辑评论后，再次 @ 了原文中已有的人，对方收到重复通知
- **复现步骤**：评论 "@张三 请查看" → 编辑为 "@张三 请尽快查看"
- **预期**：仅通知新增 @ 的人，已通知过的不重复通知
- **实际**：每次编辑都重新解析 @ 列表并全员通知

### Bug #6：分页导出 Excel 仅导出当前页（中等）
- **模块**：统计报表 - 导出
- **现象**：点击"导出全部"时只导出了当前分页的数据，而非全部数据
- **复现步骤**：任务列表共100条，当前在第1页（20条），点击导出
- **预期**：导出全部 100 条任务数据
- **实际**：仅导出当前页的 20 条数据

### Bug #7：附件24小时过期时间计算有误（中等）
- **模块**：附件上传
- **现象**：上传生成的下载链接实际23小时后即失效
- **复现步骤**：上传一个附件，记录时间，23小时后再次访问下载链接
- **预期**：链接在24小时内有效
- **实际**：使用了 UTC 时间计算但展示用了本地时间，实际过期早了1小时（UTC+8）

### Bug #8：子任务完成校验仅前端（中等）
- **模块**：任务管理 - 父子任务
- **现象**：子任务未全部完成时，通过 API 可直接将父任务标记为完成
- **复现步骤**：父任务有3个子任务，其中1个未完成 → 直接调用 API 标记父任务为完成
- **预期**：后端校验所有子任务已完成
- **实际**：校验逻辑仅在前端，API 未拦截

### Bug #9：项目删除30天物理删除的定时任务有遗漏（中等）
- **模块**：项目管理 - 软删除
- **现象**：恰好第30天时系统维护期间停机了2小时，部分项目未被物理删除
- **复现步骤**：在物理删除时间点关停服务2小时再启动
- **预期**：启动后补充执行未完成的删除
- **实际**：定时任务仅检查 `deleted_at <= NOW() - 30天`，使用精确时间点比较，错过了停机窗口的批次

### Bug #10：BigInt 精度丢失（轻微）
- **模块**：任务 ID - JavaScript 前端
- **现象**：后端使用 int64 作为任务ID（雪花算法），前端 JSON 解析后最后几位变成0
- **复现步骤**：创建任务获得 ID 如 `7896541230123456789`，前端显示为 `7896541230123456000`
- **预期**：正确显示大整数 ID
- **实际**：JavaScript Number 类型最大安全整数为 2^53-1，超出了就丢失精度

## 测试用例建议

### 功能测试
1. 项目 CRUD：正常和异常（日期倒挂、空名称、超长描述等）
2. 任务状态流转：合法和非法路径的全部组合
3. 评论：@功能、编辑/删除权限验证
4. 附件：格式校验、大小限制、过期机制
5. 看板：拖拽操作、筛选过滤

### 接口测试
1. 所有 API 端点参数边界（空值、类型错误、超长字符串、特殊字符）
2. 状态流转规则校验
3. WebSocket 断线重连场景
4. 并发指派冲突

### 性能测试
1. 大批量任务（1000+）看板加载
2. WebSocket 广播压力（100人同时在同一个看板操作）
3. 统计报表大数据量导出

### 安全测试
1. 越权操作（观察者尝试删除任务）
2. API 版本兼容性
3. 附件上传恶意文件（伪装成允许格式的病毒文件）
